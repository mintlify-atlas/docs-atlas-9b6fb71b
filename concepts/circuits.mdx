## What are circuits?

Circuits in EventConnect are computational programs written in Circom that define the logic for zero-knowledge proofs. They specify what needs to be proven without revealing the underlying private data.

<Info>
Think of circuits as constraints that inputs must satisfy. If all constraints are met, the proof is valid.
</Info>

## EventConnect's main circuit

Here's the complete main circuit for EventConnect:

```circom circuits/circuits/main.circom
pragma circom 2.0.0;
include "../../node_modules/circomlib/circuits/eddsaposeidon.circom";

template Sample() {
    // public
    signal input M;
    signal input Ax;
    signal input Ay;
    // private
    signal input S;
    signal input R8x;
    signal input R8y;

    component verifier = EdDSAPoseidonVerifier();

    verifier.enabled <== 1;
    verifier.Ax <== Ax;
    verifier.Ay <== Ay;
    verifier.S <== S;
    verifier.R8x <== R8x;
    verifier.R8y <== R8y;
    verifier.M <== M;
}

component main { public [M, Ax, Ay] } = Sample();
```

### Circuit components breakdown

<Accordion title="Signal declarations">
  Signals are the inputs and outputs of your circuit:
  
  **Public signals** (visible to verifier):
  - `M`: The message that was signed
  - `Ax`, `Ay`: Public key coordinates
  
  **Private signals** (hidden from verifier):
  - `S`: Signature scalar component
  - `R8x`, `R8y`: Signature random point coordinates
</Accordion>

<Accordion title="Component instantiation">
  ```circom
  component verifier = EdDSAPoseidonVerifier();
  ```
  
  This creates an instance of the EdDSA verifier from circomlib, which handles the cryptographic verification logic.
</Accordion>

<Accordion title="Constraint assignments">
  ```circom
  verifier.Ax <== Ax;
  ```
  
  The `<==` operator assigns values AND generates constraints. These constraints ensure the inputs satisfy the EdDSA verification equation.
</Accordion>

<Accordion title="Main component">
  ```circom
  component main { public [M, Ax, Ay] } = Sample();
  ```
  
  Declares which signals are public (all others are private by default).
</Accordion>

## Signature rollup circuit

For verifying multiple signatures, EventConnect uses a parameterized circuit:

```circom
template Sample(n) {  // n = number of signatures to verify
    // public
    signal input M;
    signal input Ax[n];
    signal input Ay[n];
    // private
    signal input S[n];
    signal input R8x[n];
    signal input R8y[n];

    component verifier[n];
    for (var i = 0; i < n; i++) {
        verifier[i] = EdDSAPoseidonVerifier();
        verifier[i].enabled <== 1;
        verifier[i].Ax <== Ax[i];
        verifier[i].Ay <== Ay[i];
        verifier[i].S <== S[i];
        verifier[i].R8x <== R8x[i];
        verifier[i].R8y <== R8y[i];
        verifier[i].M <== M;
    }
}

component main { public [M, Ax, Ay] } = Sample(3);
```

<Tip>
The parameter `n` must be set at compile time. To verify a different number of signatures, you need to recompile the circuit with a different value.
</Tip>

### How rollup works

<Steps>
  <Step title="Array inputs">
    Input signals become arrays: `Ax[n]`, `Ay[n]`, `S[n]`, etc.
  </Step>
  <Step title="Component array">
    Create `n` verifier components, one for each signature.
  </Step>
  <Step title="Loop verification">
    Each verifier checks one signature against the same message `M`.
  </Step>
  <Step title="Constraint aggregation">
    All verifiers must pass for the proof to be valid.
  </Step>
</Steps>

## Witness generation

The witness is an intermediate computation used in proof generation:

```typescript circuits/src/client.ts
// Calculate witness from inputs
const inputs = {
  M,
  Ax,
  Ay,
  S,
  R8x,
  R8y,
};

const wtns = await this.calculator.calculateWTNSBin(inputs, 0);
```

### What is a witness?

The witness contains:
1. All input signals (public and private)
2. All intermediate signals computed by the circuit
3. All output signals

<Info>
The witness proves you know valid inputs. The ZKP proves the witness is correct without revealing it.
</Info>

## Proving and verification

### Generate a proof

```typescript
const { proof } = await snarkjs.groth16.prove(this._zkey, wtns);

return {
  a: [proof.pi_a[0], proof.pi_a[1]],
  b: [proof.pi_b[0].reverse(), proof.pi_b[1].reverse()],
  c: [proof.pi_c[0], proof.pi_c[1]]
};
```

### Verify on-chain

```solidity contracts/contracts/ZkApp.sol
function verify(
    uint256[7] memory publicSignals,
    Proof memory proof
) public view returns (bool) {
    bool result = IVerifier(verifier).verifyProof(
        proof.a,
        proof.b,
        proof.c,
        publicSignals
    );
    return result;
}
```

<Tip>
The public signals array length must match your circuit's public inputs. For rollup with 3 signatures: `[M, Ax[0], Ax[1], Ax[2], Ay[0], Ay[1], Ay[2]]` = 7 elements.
</Tip>

## Circuit constraints

Understanding constraints helps optimize circuit design:

### Constraint types

<CardGroup cols={2}>
  <Card title="Equality constraints" icon="equals">
    `a === b` - Ensures two signals are equal
  </Card>
  <Card title="Multiplication constraints" icon="xmark">
    `c <== a * b` - Fundamental R1CS constraint
  </Card>
  <Card title="Addition (free)" icon="plus">
    `c <== a + b` - No additional constraints needed
  </Card>
  <Card title="Custom constraints" icon="code">
    Complex operations built from basic constraints
  </Card>
</CardGroup>

### EdDSA constraint breakdown

The `EdDSAPoseidonVerifier` uses approximately:

- **Point addition**: ~150 constraints per operation
- **Scalar multiplication**: ~1,000 constraints
- **Poseidon hash**: ~300 constraints
- **Total per signature**: ~1,500 constraints

## Circuit optimization

### Reduce constraint count

```circom
// ❌ Less efficient: Multiple multiplications
signal temp1 <== a * b;
signal temp2 <== temp1 * c;
signal result <== temp2 * d;

// ✅ More efficient: Combine operations
signal ab <== a * b;
signal cd <== c * d;
signal result <== ab * cd;
```

### Batch operations

```circom
// Process multiple items in parallel
component verifier[n];
for (var i = 0; i < n; i++) {
    verifier[i] = EdDSAPoseidonVerifier();
    // Configure verifier[i]...
}
```

<Note>
Parallel operations in circuits don't reduce constraints but improve proving efficiency.
</Note>

## Custom circuit templates

You can extend EventConnect with custom logic:

```circom
template CustomVerification(n) {
    signal input M;
    signal input Ax[n];
    signal input Ay[n];
    signal input S[n];
    signal input R8x[n];
    signal input R8y[n];
    
    // Additional custom inputs
    signal input threshold;  // Minimum required signatures
    signal output validCount;
    
    component verifier[n];
    component counter = Counter(n);
    
    for (var i = 0; i < n; i++) {
        verifier[i] = EdDSAPoseidonVerifier();
        // ... configure verifier
        counter.inputs[i] <== verifier[i].valid;
    }
    
    validCount <== counter.count;
    // Assert threshold met
    component thresholdCheck = GreaterEqThan(32);
    thresholdCheck.in[0] <== validCount;
    thresholdCheck.in[1] <== threshold;
    thresholdCheck.out === 1;
}
```

## Circomlib components

EventConnect leverages circomlib for standard operations:

### Commonly used components

```circom
include "circomlib/circuits/comparators.circom";
include "circomlib/circuits/bitify.circom";
include "circomlib/circuits/poseidon.circom";
include "circomlib/circuits/eddsaposeidon.circom";

// Use comparators
component lt = LessThan(32);
lt.in[0] <== a;
lt.in[1] <== b;
lt.out === 1;  // Assert a < b

// Use Poseidon hash
component hasher = Poseidon(2);
hasher.inputs[0] <== field1;
hasher.inputs[1] <== field2;
signal hash <== hasher.out;
```

## Debugging circuits

### Common errors

<AccordionGroup>
  <Accordion title="Error: Constraint doesn't match">
    **Cause**: Input values don't satisfy circuit constraints.
    
    **Solution**: Verify inputs are correct and match expected ranges.
    
    ```bash
    # Enable verbose output
    snarkjs groth16 prove circuit.zkey witness.wtns proof.json public.json --verbose
    ```
  </Accordion>
  
  <Accordion title="Error: Not same number of signals">
    **Cause**: Input JSON doesn't match circuit signal declarations.
    
    **Solution**: Ensure all required signals are provided with correct names.
  </Accordion>
  
  <Accordion title="Error: Value too large for field">
    **Cause**: Input exceeds Baby Jubjub field prime.
    
    **Solution**: Validate inputs are within field bounds before passing to circuit.
    
    ```typescript
    const FIELD_SIZE = BigInt("21888242871839275222246405745257275088548364400416034343698204186575808495617");
    if (value >= FIELD_SIZE) {
      throw new Error("Value exceeds field size");
    }
    ```
  </Accordion>
</AccordionGroup>
